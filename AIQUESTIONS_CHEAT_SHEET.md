- prompt: Describe the layered architecture of the Gear4Music PHP inventory system and explain why separating Repository, Service, Controller, and Presenter into distinct layers matters for business value.
  question_set: gear4music interview cheat sheet
  answer: The Gear4Music system uses a four-layer architecture: Repository (handles data access), Service (applies business rules), Controller (orchestrates requests), and Presenter (formats output). Each layer has a single responsibility. This matters because when requirements change—for example, moving from in-memory to MySQL storage, or adding a new currency conversion rule—changes are isolated to the affected layer. Service logic doesn't know or care whether data comes from memory or a database. Controllers stay thin and don't leak business logic. This design reduces the blast radius of change and makes the codebase safer to evolve.
  explanation: This question probes whether the candidate understands that architectural layering is not busywork but a concrete business practice—reducing risk when adding features, switching infrastructure, or fixing bugs.

  explanation: This demonstrates understanding of responsibility boundaries and why business logic belongs in a domain service, not scattered across infrastructure or presentation layers.

- prompt: Explain why Gear4Music uses an interface (InventoryInterface) even though there is currently only one repository implementation. How does this design decision support future work?
  question_set: gear4music interview cheat sheet
  answer: Using an interface with one implementation looks like over-engineering at first but provides concrete benefits: (1) The service depends on a contract, not a concrete class. Swapping from InMemoryProductRepository to a MySQLProductRepository later requires only a new repository class; service code doesn't change. (2) Unit testing becomes simpler—you create a mock or test double that implements InventoryInterface without a database. (3) It signals to the team that storage is swappable by design. If you hardcoded the repository class into the service, the next engineer might assume storage and logic are tightly coupled. The interface makes the boundary explicit. This is pragmatic design—you're not gold-plating, you're making one specific thing very easy to change (storage layer) because you know it will change.
  explanation: This probes understanding of abstraction as a tool for risk reduction and easier testing—not as pure ceremony.

- prompt: Explain how the Gear4Music design allows DigitalProduct to reuse Product behavior while overriding physical fields and adding digital-specific rules. Why is this better than adding optional fields to Product?
  question_set: gear4music interview cheat sheet
  answer: DigitalProduct extends Product and overrides behavior: it always returns true for visibility (no country restrictions), always true for availability (instant delivery, no stock), and zero for weight/volume (no shipping). This keeps the base Product class focused—properties and methods that apply to all products. Digital-specific overrides live in DigitalProduct. If you instead added optional fields (isDigital, weight_override, etc.) to Product, the base class becomes bloated and confusing. Logic like "if digital, ignore country" spreads through service methods. The inheritance approach is cleaner: each product type is responsible for its own semantics. The Service treats all products polymorphically (all respond to getName(), getPrice(), isVisible(), etc.) without caring if it is physical or digital.
  explanation: This tests understanding of inheritance as a tool for clean polymorphism and domain-driven design.

- prompt: In the Gear4Music system, to add an admin-only endpoint that lists all products regardless of country visibility, where should authorization checks live and why?
  question_set: gear4music interview cheat sheet
  answer: Authorization should be checked at the boundary—in middleware or a controller policy layer—not deep in the service or presenter. The reason: authorization is a cross-cutting concern that applies to HTTP routes, not business logic. If you embed role checks inside service methods, you couple domain logic to HTTP concepts and it becomes hard to reuse the service for non-HTTP contexts (CLI, scheduled jobs, etc.). A middleware or policy layer intercepts the request, checks "Is this user an admin?", and either grants access or rejects it before business logic runs. If access is granted, the rest of the code—service, repository, presenter—proceeds normally. The service doesn't know or care about roles; it just applies business rules. This keeps concerns clean and testable.
  explanation: This evaluates understanding of cross-cutting concerns and layered architecture principles.

- prompt: The Gear4Music proof of concept uses in-memory storage and scales linearly. Explain the tradeoff the architect made and what changes would be needed to support 1000+ products in production.
  question_set: gear4music interview cheat sheet
  answer: The tradeoff: the proof of concept optimized for speed of delivery and clarity of code, not scale. With in-memory storage, each query scans all products and filters in code—O(n) complexity. For 1000+ products, this becomes slow and memory-intensive. To scale: (1) Replace in-memory repository with queryable persistence (MySQL, PostgreSQL). (2) Add indexing on frequently filtered fields (country, language, condition). (3) Implement pagination to avoid loading all products. (4) Add caching for high-frequency reads (e.g., "show all guitars in UK with price in GBP"). The architectural design helps: you only need to replace the Repository implementation. Service, Controller, and Presenter logic stay mostly the same. This is pragmatic: you prove the design early with in-memory data, then scale the storage layer without rewriting business logic.
  explanation: This probes realistic thinking about POC vs. production and how good architecture makes scaling less painful.

- prompt: In Gear4Music, explain why the Presenter should not contain business logic and what would go wrong if it did.
  question_set: gear4music interview cheat sheet
  answer: The Presenter's job is to transform domain objects into a transport format (JSON), not to calculate, filter, or decide business rules. If you put logic like "if currency is USD, apply 15% markup" into the Presenter, you create hidden business rules that are hard to test (require mocking JSON serialization), easy to duplicate (other endpoints also check currency), and expensive to change (logic is scattered). More importantly, you hide policy behind presentation—the next developer doesn't realize that markup exists because it is buried in JSON formatting code. By keeping the Presenter dumb (only format what the Service provides), all business decisions are visible in the Service, testable as business logic, and reusable across entry points. If you later build a CSV export or a CLI output, you reuse the Service and Presenter layers; the business logic does not get reimplemented.
  explanation: This tests understanding of separation of concerns and why presentation is a thin layer, not a logic layer.

- prompt: The Gear4Music system uses enums for product condition (new/used/refurbished) in the domain model. Explain the design benefits of this choice over using plain strings.
  question_set: gear4music interview cheat sheet
  answer: Enums provide type safety and typo prevention. If condition is a string, a mistake like `condition = "refurbished"` (typo: two 'r's, missing 'e') compiles fine and silently becomes an invalid state. Enum values are defined once; typos are caught at compile time. Enums also improve documentation—the choices are explicit and can be introspected. In the Gear4Music design, DigitalProduct always overrides condition to "new", which is natural; you can assert that `condition == Condition::NEW` with full type confidence. Without enums, you write string comparisons everywhere, logic gets fragile, and future maintenance is error-prone. Enums are a form of domain-driven design: the domain concept "condition" is modeled as a real type, not as stringly-typed data.
  explanation: This evaluates understanding of type safety as a scalable design practice that reduces bugs and improves maintainability.

With the interface, adaptation is a isolated task.

- prompt: Explain why the Gear4Music architect chose to validate country, language, and currency at the request boundary and the risk of skipping this step.
  question_set: gear4music interview cheat sheet
  answer: Validating at the boundary (Controller/Service input handling) means invalid data never enters the system. If a user passes country="INVALID", you reject it immediately with a clear error. Risk of skipping: if invalid data reaches the Service, it may silently cause wrong behavior—filtering logic fails silently, comparisons don't match, data is returned inconsistently. Worse, bugs may be environment-specific (invalid codes work in testing but not production) or intermittent (cache hits miss the invalid case). Boundary validation also provides security: you prevent over-posting (extra fields), injection attacks, and resource exhaustion (very large inputs). By validating early, you ensure downstream code can assume data is already safe, simplifying logic and testing.
  explanation: This tests understanding of defensive programming and input validation as a scaling and security practice.
